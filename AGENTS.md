# CleaningApp — Agent Guidelines

> Last updated: 2026-05-10 | Reflects current codebase state

## Project Overview

iOS app (Swift 6.0, SwiftUI, iOS 18.0+) using a RIB-inspired architecture (Router–Interactor–Builder + Presenter + View). The project file is generated by **XcodeGen** from `project.yml`. Never edit `.xcodeproj` directly — edit `project.yml` and regenerate.

**Current Development Status:** Early-mid development (~35% MVP complete). Onboarding flow is production-ready, data layer complete, but core features (scheduling algorithm, home view with daily tasks, gamification, notifications) are pending implementation.

## Architecture — RIB Pattern

Every feature follows this structure:

```
FeatureName/
├── Presentation/
│   ├── FeatureNameView.swift      # SwiftUI View only — no logic
│   └── FeatureNamePresenter.swift # @Observable class — bridges View ↔ Interactor/Router
└── RIB/
    ├── FeatureNameBuilder.swift   # Wires dependencies and builds the View
    ├── FeatureNameInteractor.swift# Protocol definition + CoreInteractor conformance
    └── FeatureNameRouter.swift    # Protocol definition + CoreRouter conformance
```

**Key rules:**
- `CoreInteractor` is the single concrete interactor — it conforms to every feature's interactor protocol via extensions in the feature's own `FeatureInteractor.swift` file.
- `CoreRouter` follows the same pattern for routing.
- Presenter holds protocols as existentials: `private let interactor: any FeatureInteractor`.
- `Builder` builds and returns the fully wired `View` — it receives interactor/router from `CoreBuilder`.
- Business logic lives in `Services/`, not in Interactors or Presenters.

## Code Style

### Formatting (`.swiftformat`)
- Indent: **4 spaces**
- Swift version: 6.0
- Import grouping: `@testable` imports last
- Collection wrapping: break before first element in multi-line collections

### Imports
- System/Apple frameworks first (`Foundation`, `SwiftUI`, `SwiftData`, `UIKit`), then SPM packages alphabetically.
- Only import what the file directly uses — no wildcard or redundant imports.
- SwiftLint enforces `unused_import` via analyzer rules.

### Naming
- **Types**: `UpperCamelCase` with descriptive suffix: `View`, `Presenter`, `Interactor`, `Router`, `Builder`, `Manager`, `Repository`, `Mapper`, `Entity`, `State`.
- **Functions**: `lowerCamelCase` verbs. UI action handlers prefixed with `on`: `onGetStartedPressed()`, `onFinishButtonPressed()`.
- **Enum cases**: `lowerCamelCase`.
- **Files**: One type per file, named exactly after the type.

### Access Control
- All properties default to `private` unless they must be visible externally.
- Use `private(set)` for properties that are read externally but written internally.
- All classes are `final` unless inheritance is explicitly required.
- No `public` or `open` — this is a single-module app.
- `private` extensions for file-scoped helpers (e.g., `private extension UserDefaults`).

### `@Observable` & State Management
- Presenters are `@Observable @MainActor final class`.
- Views hold presenters as `@State var presenter: FeaturePresenter` — never `@StateObject`.
- No `@Published` anywhere — use `@Observable` exclusively.
- `@Binding` only when a child view must mutate parent state.
- All types (classes, structs, protocols) are annotated `@MainActor`.

### Concurrency
- Everything runs on `@MainActor` — protocols, structs, classes all carry the annotation.
- No `async/await` or `Task` in existing code — service calls are synchronous and `throws`.
- Do not introduce `async/await` without discussing it first; the pattern isn't established yet.

### MARK Comments
Use `// MARK: - SectionName` (with space after em-dash) to structure every type. Standard sections:
```swift
// MARK: - TypeName       ← above the type declaration itself

// MARK: - Properties
// MARK: - Init
// MARK: - Body           ← in Views
// MARK: - Actions
// MARK: - Navigation
// MARK: - Methods
// MARK: - Preview        ← above #Preview blocks
```

### Error Handling
- Service methods use `throws` (not `async throws`).
- `try!` only for app-startup fatal failures (e.g., `ModelContainer` initialisation).
- Force-unwrap (`!`) only on `DependencyContainer.resolve()` where the dependency is guaranteed registered at startup.
- No `try!` or `!` in feature code or tests.

### Conditional Compilation
```swift
#if MOCK
// mock-only code
#elseif DEV
// dev-only code
#else
// production code
#endif

#if DEV || MOCK
// shown in non-production builds
#endif
```

### SwiftData Layer
- Persistence entities: `@Model final class XxxEntity` in `Models/Entities/`.
- Domain models: plain `struct` in `Models/Domain/` — no framework imports.
- Mappers: `enum XxxMapper` with `static func toDomain(_:)` and `static func toEntity(_:)`.
- Repositories: `protocol XxxRepository` + `MockXxxRepository` (in-memory) + `SwiftDataXxxRepository` — all in one file.
- Managers: `@MainActor final class XxxManager` wrapping `any XxxRepository` — the only type callers interact with.

**Special:** `FrequencyEncoder` — enum-based encoder for `Frequency` enum to String (SwiftData doesn't support enums with associated values directly).

### SwiftLint Rules (opt-in)
- `empty_count` — use `.isEmpty` not `== 0`
- `closure_spacing` — space inside closure braces
- `force_unwrapping` — warns on `!` (treat as error in feature code)
- `unused_import` — via analyzer

## Testing

- Framework: **Swift Testing** (`import Testing`), not XCTest.
- Tags are defined in `CleaningAppTests/Tags.swift` as `extension Tag`.
- Available tags: `.home`, `.rooms`, `.settings` (feature); `.adding`, `.deleting`, `.editing`, `.navigation` (behavior); `.critical`, `.slow` (priority).
- Mocks live in `CleaningAppTests/Shared/Mocks/`.
- Use the `CleaningApp - Mock` scheme for tests (in-memory `MockRepository` implementations, no SwiftData).
- Test method names: descriptive, use underscores to separate sections if needed: `test_save_persistsRoom`.

**Test Coverage:** Excellent for implemented features. 20 test files covering:
- All persistence layer (repositories, mappers, managers)
- Onboarding flow (5 test files + snapshot tests)
- Total: ~93 app files, 20 test files

## Adding a New Feature

1. Create folder: `Core/FeatureName/Presentation/` and `Core/FeatureName/RIB/`
2. Create `FeatureNameInteractor.swift` — define `@MainActor protocol FeatureNameInteractor` and add `extension CoreInteractor: FeatureNameInteractor {}`
3. Create `FeatureNameRouter.swift` — same pattern with `CoreRouter`
4. Create `FeatureNamePresenter.swift` — `@Observable @MainActor final class` holding `any FeatureNameInteractor` and `any FeatureNameRouter`
5. Create `FeatureNameView.swift` — `@MainActor struct` with `@State var presenter: FeatureNamePresenter`
6. Create `FeatureNameBuilder.swift` — `@MainActor struct` conforming to `Builder`
7. Register the tab/route in `CoreBuilder.swift` and wire the interactor/router in the builder
8. Update `project.yml` if new files aren't picked up, then run `xcodegen generate`
